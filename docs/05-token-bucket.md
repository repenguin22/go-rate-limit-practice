# Stage 4: トークンバケット (Token Bucket)

実装: [tokenbucket.go](../ratelimit/tokenbucket.go) / テスト: [tokenbucket_test.go](../ratelimit/tokenbucket_test.go)

## 1. アイデア: 「数える」のをやめて「貯めて使う」

ここまでの3方式はすべて「過去のリクエストを数える」発想だった。
トークンバケットは視点を反転させ、「**未来に使える権利を貯める**」と考える。

```
              補充: window あたり limit 個の一定速度
              (例: 10個/分 = 6秒に1個ずつ)
                      │
                      ▼
              ┌───────────────┐
              │  ● ● ● ●      │ ← バケツ(容量 = burst)
              │    ● ● ●      │    貯まるのは burst 個まで。溢れた分は捨てる
              └───────┬───────┘
                      │
    リクエスト ──▶ トークンを1個消費して通過。なければ拒否
```

```
ルール:
1. バケツには window あたり limit 個の一定速度でトークンが補充される
2. トークンは burst 個(容量)までしか貯まらない
3. リクエストはトークンを1個消費して通過。トークンがなければ拒否
```

この設計から2つの性質が**同時に**得られる:

- **平均レートの保証**: 長期的には補充速度(limit/window)以上は絶対に通らない
- **バーストの許容**: しばらく使っていなければトークンが貯まっており、
  貯金を一気に使える(最大 burst 回)

「普段は静かだが、たまにまとめて叩く」——正当なクライアントの多くはこういう
トラフィックパターンを持つ。ウィンドウ系がこれを「攻撃」と区別できないのに対し、
トークンバケットは**平均さえ守っていればバーストを歓迎する**。この寛容さが
実務で選ばれ続ける理由(AWS API、Stripe、`golang.org/x/time/rate` すべてこの方式)。

## 2. 実装の読みどころ

### 2.1 補充ゴルーチンは要らない —— 遅延補充

素朴に考えると「6秒ごとにトークンを足すゴルーチン」が必要に見えるが、
key が100万あれば100万個のタイマーになり破綻する。

実際には**アクセスされた瞬間に、前回からの経過時間分をまとめて補充**すればよい:

```go
type tbState struct {
	tokens float64   // 現在のトークン残量(小数になりうる)
	last   time.Time // 最後に補充計算をした時刻
}

// Allow 内
st.tokens = math.Min(tb.burst, st.tokens+tb.tokensIn(now.Sub(st.last)))
st.last = now
```

「状態 + 最終更新時刻」を持ち、参照時に時間経過を織り込む——Stage 1 の
「次のアクセス時に遅延評価でリセット」と同じ発想の一般化で、時間発展する
状態を扱うときの定石。

トークンが `float64` なのもポイント。「6秒で1個」の途中の「3秒で0.5個」を
表現できるため、補充が離散的なステップではなく連続的な流れになる。

### 2.2 浮動小数点の罠: 除算は最後に

今回のステージで実際にテストを割ったバグがこれ。最初の実装は
「1ナノ秒あたりの補充率」を先に計算していた:

```go
rate := float64(limit) / float64(window)     // 10/6e10 → 誤差を含む
refill := float64(elapsed) * rate            // 6e9 × 1.666..e-10 = 0.9999999999999999
```

`limit/window` は2進浮動小数点で正確に表現できないことが多く、
「ちょうど6秒待ったのにトークンが 0.9999999999999999 個しかない」が起きる。
**掛け算を先に、除算を最後に**すれば整数値どうしの演算が正確なまま保たれる:

```go
refill := float64(elapsed) * float64(limit) / float64(window) // 6e10/6e10 = 1.0 ちょうど
```

時間×レートの計算をするコードすべてに通じる教訓
(`x/time/rate` も同じ理由で除算を遅らせる書き方をしている)。

### 2.3 レートとバーストが独立に設定できる

ウィンドウ系は「10回/分」という1つの設定しかなかったが、トークンバケットは
**平均レート**と**瞬間許容量**を別々に決められる:

```go
// 平均1回/秒。ただし瞬間的には5回まで OK
tb, _ := ratelimit.NewTokenBucket(1, time.Second, ratelimit.WithBurst(5))
```

`TestTokenBucket_CustomBurst` がこの挙動を検証している。運用では
「平均は厳しく、バーストは実測のピークに合わせて」のように別々に
チューニングできるのが大きい。

### 2.4 RetryAfter は「不足分が貯まるまで」

```go
retry := tb.durationFor(1 - st.tokens) // 残り(1 - tokens)個が貯まる時間
```

固定ウィンドウのような「ウィンドウ末尾まで待て」ではなく、必要な分だけの
待ち時間が正確に出る。`TestTokenBucket_RetryAfter` は
「RetryAfter どおり待てば必ず通る」ことまで検証している——この性質は
クライアントへの案内として重要(待たせたのにまだ拒否、が起きない)。

### 2.5 掃除の条件は「満タンに戻ったか」

```go
if st.tokens+tb.tokensIn(now.Sub(st.last)) >= tb.burst {
	delete(tb.states, key)
}
```

満タンのバケツは**新規 key と区別がつかない**ので、消しても判定に影響しない。
Stage 3 の「2ウィンドウ待つ」と同様、「いつ消してよいか」はアルゴリズムごとに
論理的に導く必要がある。ここでの答えは「状態が初期状態と等価になったとき」。

## 3. 弱点と注意点

- **burst の設定を誤ると大きく通る**: burst=10000 にすれば瞬間1万リクエストが
  通る。バックエンドの瞬間耐性(コネクション数、キュー深さ)と相談して決める
- **「厳密に N回/分」の保証はない**: 任意の60秒間で見ると最大 burst + limit 回
  通りうる(満タンから使い切り+その間の補充)。ウィンドウ的な上限が
  契約上必要なら(課金APIのクォータなど)ウィンドウ系を使う
- **float64 の精度**: 極端に大きな limit × 長い window では誤差が蓄積しうる。
  `x/time/rate` は同じ構造をより注意深い数値処理で実装している(Stage 7 で読む)

## 4. 特性まとめ

| 観点 | 評価 |
|------|------|
| 時間計算量 | O(1) / リクエスト |
| 空間計算量 | key あたり float64 + 時刻のみ(最小クラス) |
| 平均レート | 厳密に limit/window 以下 |
| バースト | burst 個まで明示的に許容(独立にチューニング可能) |
| RetryAfter | 厳密 |
| 1判定の速度 (M1 Pro) | ~290ns、アロケーションゼロ |
| 向いている場面 | 一般APIの流量制限ほぼ全部。迷ったらこれ |
| 向かない場面 | 「任意の期間で厳密に N 回以下」の契約的保証が必要な場面 |
