# Stage 3: スライディングウィンドウカウンタ (Sliding Window Counter)

実装: [slidingwindowcounter.go](../ratelimit/slidingwindowcounter.go) / テスト: [slidingwindowcounter_test.go](../ratelimit/slidingwindowcounter_test.go)

## 1. アイデア: ログを持たずに「滑るウィンドウ」を近似する

ここまでの2方式はきれいなトレードオフの関係にあった:

- 固定ウィンドウ: カウンタ1個で済むが、境界バーストを許す
- スライディングウィンドウログ: 厳密だが、タイムスタンプを全部持つ

両者のいいとこ取りをしたい。key あたり**カウンタ2個**(前ウィンドウと現ウィンドウ)
だけ持ち、「滑るウィンドウ内の件数」を**推定**する:

```
window=1分, いま 00:01:20(現ウィンドウ開始から20秒 = 1/3 経過)

     前ウィンドウ (00:00〜00:01)      現ウィンドウ (00:01〜00:02)
    │        prev = 9件           │      curr = 4件      │
    ├────────┼─────────────────── ┼──────────────┼───────┤
             │                                   │
             00:00:20 ◀──── 滑るウィンドウ ────▶ 00:01:20(いま)
             └── 前ウィンドウと 2/3 重なっている ──┘

  推定カウント = prev × (前ウィンドウとの重なり率) + curr
              = 9 × 2/3 + 4 = 10
```

重なり率は `1 - 経過率` で求まり、現ウィンドウが進むにつれ 1 → 0 へ線形に減衰する。
これは「**前ウィンドウのリクエストは期間中に均等に分布していた**」という仮定に
基づく近似。実際の分布が偏っていれば誤差が出るが(後述)、実用上は十分良い。
Cloudflare が全エッジで採用し、実トラフィックの 99.997% を正しく判定できたと
報告している方式。

```
ルール:
1. リクエストが来たら、ウィンドウの切り替わりを処理(curr → prev へシフト)
2. estimated = prev × weight + curr を計算
3. estimated < limit なら curr++ して許可、でなければ拒否
```

## 2. 実装の読みどころ

### 2.1 状態はカウンタ2個に圧縮された

```go
type swcState struct {
	windowStart time.Time // 現ウィンドウの開始時刻
	prev        int       // 前ウィンドウの確定カウント
	curr        int       // 現ウィンドウのカウント
}
```

Stage 2 の `[]time.Time` がこれだけになった。limit がいくつでも key あたりの
メモリは固定。ここが本方式の存在意義。

### 2.2 ウィンドウのシフト処理

ウィンドウが切り替わったとき、「1つだけ進んだ」のか「2つ以上飛んだ」のかで
処理が違う。ここはバグを埋めやすいポイント:

```go
if st.windowStart.Equal(start.Add(-sc.window)) {
	st.prev = st.curr // 1つ進んだ: 現カウントが「前ウィンドウ」になる
} else {
	st.prev = 0       // 2つ以上飛んだ: どちらのカウントももう無関係
}
st.curr = 0
st.windowStart = start
```

もし常に `prev = curr` としてしまうと、1時間アクセスがなかったユーザーの
1時間前のカウントが「前ウィンドウ」として重み付けされてしまう。
`TestSlidingWindowCounter_FullResetAfterGap` がこの分岐を検証している。

### 2.3 推定と判定は3行

```go
elapsed := now.Sub(start)
weight := 1 - float64(elapsed)/float64(sc.window)
estimated := float64(st.prev)*weight + float64(st.curr)
```

境界バーストがなぜ防がれるかもこの式から読める: 次ウィンドウの先頭
(elapsed=0)では weight=1.0 なので、**前ウィンドウのカウントがそのまま全部
持ち越される**。固定ウィンドウのように突然ゼロに戻ることがない。
`TestSlidingWindowCounter_MitigatesBoundaryBurst` は Stage 1・2 と同じ
攻撃シナリオ(00:00:59 に10回 → 00:01:00 に10回)で、後半が全拒否になることを
確認している。

### 2.4 RetryAfter は方程式を解いて出す

拒否したとき、いつになったら通るのか。weight が時間とともに線形に減ることを
利用して、`prev*(1 - e/window) + curr < limit` を経過時間 `e` について解く:

```go
e := float64(sc.window) * float64(st.prev+st.curr-sc.limit) / float64(st.prev)
```

面白いのは、この方式では枠が「**連続的に、少しずつ**」回復すること。
前ウィンドウで10回使い切った場合(limit=10, window=60s)、次ウィンドウでは
約6秒ごとに1枠ずつ滲み出すように回復していく。固定ウィンドウの
「1分待ったら一気に10枠」とは対照的で、結果的に流量が滑らかになる。
`TestSlidingWindowCounter_GradualRecovery` で実際の数値を追える。

なお `curr >= limit` の場合は現ウィンドウ内では絶対に回復しないので、
次ウィンドウ開始までの時間を返す(こちらは近似)。

### 2.5 掃除は「2ウィンドウ」待つ必要がある

```go
if !now.Before(st.windowStart.Add(2 * sc.window)) {
	delete(sc.states, key)
}
```

固定ウィンドウでは「ウィンドウが終わったら削除してよい」だったが、今回は
現ウィンドウのカウントが**次のウィンドウでも prev として参照される**。
1ウィンドウで消すと、消された key は次アクセス時に prev=0 で再作成され、
本来より多く許可してしまう。`TestSlidingWindowCounter_DeleteExpired` は
「1ウィンドウ経過の key は残り、2ウィンドウ経過の key だけ消える」ことを
検証している。

状態に寿命があるアルゴリズムでは、**掃除のタイミングは判定ロジックと同じ
くらい正しさに直結する**——これはキャッシュ設計全般に通じる教訓。

## 3. 弱点: あくまで近似

「前ウィンドウは均等分布」の仮定が外れると誤差が出る。最悪ケース:

```
limit=10, window=1分
前ウィンドウ: 00:00:00 の瞬間に10回(先頭に偏った分布)
現在: 00:01:30(weight=0.5, estimated = 10×0.5 = 5 → あと5回許可)

実際の直近60秒(00:00:30〜00:01:30)のリクエスト数は 0件。
本当は10回許可してよいのに5回しか許可されない(過剰に厳しい方向の誤差)
```

逆に、前ウィンドウの終端に偏っていた場合は過剰に緩い方向へずれる。
誤差は最大で limit の数十%になりうるが、**境界バーストのような「2倍」は
構造的に起きない**(weight=1 の瞬間は前ウィンドウ全体が乗るため)。
「厳密な保証」ではなく「実用的な流量制御」が目的なら、このコストで
この精度は極めて割が良い。

## 4. 特性まとめ

| 観点 | FixedWindow | SlidingWindowLog | **SlidingWindowCounter** |
|------|------------|------------------|--------------------------|
| key あたりメモリ | 時刻+int | O(limit) | **時刻+int×2** |
| 1判定の速度 (M1 Pro) | ~234ns | ~261ns | **~254ns** |
| 境界バースト | 最大2倍 | なし(厳密) | **実用上なし(近似)** |
| RetryAfter | ウィンドウ末尾 | 厳密 | ほぼ厳密(方程式) |
| 向いている場面 | 粗いクォータ | 少数回の高価値操作 | **一般APIの流量制限** |
