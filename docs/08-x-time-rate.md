# Stage 7: golang.org/x/time/rate を読む

比較テスト: [xtimerate_test.go](../ratelimit/xtimerate_test.go)
読む対象: [golang.org/x/time/rate](https://pkg.go.dev/golang.org/x/time/rate)([ソース](https://cs.opensource.google/go/x/time/+/master:rate/rate.go))

## 1. なぜ「本物」を読むのか

自作した5方式のうちトークンバケットには、Goの準標準ライブラリに定番実装がある。
`golang.org/x/time/rate` は net/http のプロキシ、gRPC、Kubernetes など
無数のプロダクションで使われている。これを読むと:

- 自作実装の設計判断が正しかったか答え合わせできる
- 「同じアルゴリズムでもAPI設計でこれだけ変わる」を学べる
- 実務で「作らずに済む場面」を見分けられるようになる

答え合わせの手段として、**同じ入力列に対して自作 TokenBucket と x/time/rate が
全リクエストで同じ判定を返す**ことをテストで実証した
(`TestTokenBucket_MatchesXTimeRate`)。アルゴリズムの核が同一である証拠。

## 2. API設計: 消費のしかたが3つある

自作の `Allow` は「今すぐ判定、ダメなら拒否」の一択だった。
`x/time/rate` はトークンの消費方法を3つ提供する:

```go
lim := rate.NewLimiter(rate.Every(6*time.Second), 10) // 6秒に1個補充、容量10

ok := lim.Allow()          // ① 今トークンがあるか。なければ false(拒否)
r := lim.Reserve()         //  ② 未来のトークンを予約。r.Delay() だけ待ってから実行
err := lim.Wait(ctx)       // ③ トークンが取れるまでブロック(ctx で中断可能)
```

| | 意味 | 使いどころ |
|---|---|---|
| `Allow` | 拒否する | サーバー側の保護(429を返す) |
| `Reserve` | 予約する | 自前でスケジューリングしたいとき |
| `Wait` | 待つ | クライアント側の自己制御(外部API呼び出しなど) |

ここで Stage 5 の伏線が回収される。**`Wait` は「キュー型リーキーバケット」
そのもの**——溢れない限り拒否せず、一定レートになるまで待たせる。
アルゴリズムを変えずに、APIの形だけでポリシング(Allow)と
シェーピング(Wait)を両立している。

### Reserve の実装が面白い: トークンは「負」になれる

`Reserve` は今トークンがなくても予約を成立させる。内部では
**トークン残高がマイナスになる**:

```
tokens = -2.5 …… 「未来の2.5個ぶんを前借りしている」状態
```

`Delay()` は残高が0に戻る時刻から逆算される。自作実装は
`tokens >= 1` を判定するだけだったが、残高に負を許すと
「いつなら実行できるか」の計算に自然に拡張できる。
`Wait` は「Reserve して Delay ぶん `time.Timer` で眠る」だけで実装されている。

## 3. 内部実装の答え合わせ

ソース(rate.go、500行弱)を読むと、核は自作とほぼ同じことが分かる:

| 設計判断 | 自作 TokenBucket | x/time/rate | 一致? |
|---|---|---|---|
| 状態 | `tokens float64` + `last time.Time` | 同じ(`tokens`, `last`) | ✅ |
| 補充 | アクセス時にまとめて(遅延補充) | 同じ(`advance` メソッド) | ✅ |
| 排他 | `sync.Mutex` | 同じ | ✅ |
| 端数 | float64 で連続的に持つ | 同じ | ✅ |
| 満タン頭打ち | `math.Min(burst, …)` | 同じ | ✅ |

**浮動小数点の罠まで同じだった**ことは特筆に値する。時間⇄トークンの変換
(`tokensFromDuration`)は当初 `d.Seconds() * float64(limit)` という素朴な実装で、
精度バグ([golang.org/issue/34861](https://github.com/golang/go/issues/34861))として
報告され、秒とナノ秒を分けて計算する形に修正された。われわれが Stage 4 で
踏んだ「除算と乗算の順序」と同じクラスの問題を、本家も踏んでいる。
**時間×レート計算の精度は、実装者が誰でも気をつける必要がある**ということ。

### テスト容易性: クロック注入 vs 時刻引数

x/time/rate には `Clock` インターフェースがない。代わりにほぼ全メソッドに
時刻を明示的に渡す変種がある:

```go
lim.AllowN(t, 1)      // 「時刻 t だったとしたら」の判定
lim.ReserveN(t, 1)
```

比較テストでこれを利用した——x/time/rate 側には偽クロックが要らず、
`clock.Now()` の値をそのまま渡せばよい。

| | Clock 注入(自作) | 時刻引数(x/time/rate) |
|---|---|---|
| 呼び出し側のAPI | きれい(時刻が見えない) | 引数が1個増える |
| テスト | 偽クロックを用意 | 時刻を渡すだけ |
| 誤用リスク | 低い | 過去の時刻を渡すと壊れうる |

どちらも実務で使われる正当なパターン。ライブラリの規模と利用者層で選ぶ。

### 自作にない機能: 動的再設定

`SetLimit` / `SetBurst` で運転中にレートを変更できる。障害時に手動で絞る、
負荷に応じて自動調整する(アダプティブレートリミット)といった運用の
入り口になる。変更時に既存の予約との整合を取る処理が入っているのも
読みどころ。

## 4. 逆に、x/time/rate に「ない」もの

自作パッケージの存在意義でもある:

- **key の概念がない。** `rate.Limiter` は1個で1エンティティ。ユーザーごとの
  制限には `map[string]*rate.Limiter` + ミューテックス + 掃除(ジャニター)を
  自分で書くことになる——つまり本プロジェクトで作ったものとほぼ同じ物が必要になる
- **判定のメタデータが乏しい。** `Allow()` は bool だけ。`X-RateLimit-Remaining`
  や `Retry-After` を返したければ `Reserve` を使って `Delay()` から逆算するか、
  `Tokens()` を別途呼ぶ(アトミックでない)ことになる。自作の `Result` は
  この不便への回答
- **分散対応がない。** 完全にインメモリ・単一プロセス前提。`ctx` を取るのは
  `Wait` だけで、`Allow` はエラーも返さない——「バックエンドは絶対に
  ローカル」という設計上の宣言。Stage 0 で自作インターフェースに
  `ctx` と `error` を入れた判断との対比がここで際立つ
- **トークンバケット以外がない。** 厳密なウィンドウ保証(Stage 2)や
  固定クォータ(Stage 1)が要件なら別の道具が要る

## 5. 使い分けの結論

- **単一プロセスで、自分が呼び出す側を絞る**(外部APIのレート契約を守る、
  バッチの流量制御)→ 迷わず `x/time/rate`。`Wait(ctx)` が完璧にはまる。
  車輪の再発明をしない
- **key ごとの制限、HTTPヘッダ用のメタデータ、分散**が要る → 本プロジェクトの
  ような keyed / Result / ctx+error 設計が必要になる(既製品なら
  そういう設計のライブラリを選ぶ)

性能はほぼ互角(M1 Pro): 自作 294ns/op に対し x/time/rate 244ns/op。
差分はほぼ自作側の `map[string]` 参照コストで、keyを持つ以上避けられない固定費。
