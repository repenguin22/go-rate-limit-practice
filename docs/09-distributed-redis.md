# Stage 8: 分散レートリミット (Redis)

実装: [redislimit/](../redislimit/) / テスト: [redislimit_test.go](../redislimit/redislimit_test.go)

## 1. なぜ分散版が必要か

実際のサービスはアプリサーバーを複数台並べる。インメモリのリミッターは
**各サーバーが自分のカウントしか知らない**ので、10台構成で「100回/分」を
設定すると実質1000回/分まで通ってしまう。しかもLBの振り分け次第でばらつく。

```
             ┌─ app1 (カウント: 34) ─┐
クライアント ──┤─ app2 (カウント: 33) ─├──  それぞれ独立に 100 まで許可
             └─ app3 (カウント: 33) ─┘   = 実質 300/分

             ┌─ app1 ─┐
クライアント ──┤─ app2 ─├── Redis (カウント: 100) ← 1つの真実
             └─ app3 ─┘
```

解決策は**状態を共有ストアに置く**こと。Redis が定番なのは、シングル
スレッドで原子性を作りやすく、TTL・Lua・低レイテンシが揃っているから。
`TestFixedWindow_SharedAcrossInstances` が分散版の存在意義そのものを
テストにしている: 2つのリミッターインスタンス(=2台のアプリサーバー)の
カウントが合算で効く。

インターフェースは Stage 0 の `Limiter` のまま。**ここで `ctx` と `error` を
最初から入れておいた設計が回収される**——呼び出し側(ミドルウェア)は
一切変更なしでインメモリ版と差し替えられる。

## 2. 核心①: 原子性 —— なぜ Lua スクリプトなのか

素朴な実装はこうなる:

```
count = GET key      ← ①読む
if count < limit:
    INCR key         ← ②書く
```

①と②の間に他のサーバーが割り込める。limit=100 で残り1枠のとき、
10台が同時に①を実行すると全台が「99 < 100、OK」と判断し、10リクエストが
通る——**check-and-set 競合**。インメモリ版で `sync.Mutex` が防いでいたものが、
ネットワークを挟むと防げなくなる。

Redis での答えが **Luaスクリプト**。Redis はスクリプト全体を他のコマンドを
挟まず実行する(シングルスレッドなので自然にそうなる)。つまり
**Lua スクリプトが分散環境における Mutex のクリティカルセクションに相当する**:

| インメモリ版 | Redis版 |
|---|---|
| `mu.Lock()` 〜 `mu.Unlock()` | Lua スクリプト1回の実行 |
| Goの構造体 | Redis のハッシュ |
| `clock.Now()` | `redis.call('TIME')` |
| ジャニター | `PEXPIRE`(TTL) |

スライディングウィンドウとトークンバケットのスクリプトは、インメモリ版の
`Allow` と**1対1対応**するよう書いてある。
[tokenbucket.go(Redis版)](../redislimit/tokenbucket.go) と
[tokenbucket.go(インメモリ版)](../ratelimit/tokenbucket.go) を並べて読むと、
遅延補充・容量頭打ち・「除算は最後に」まで同じ構造なのが分かる。

**固定ウィンドウだけは例外**で、ウィンドウの定義そのものが違う。
インメモリ版は壁時計に整列していた(`Truncate`。毎分0秒に全 key が一斉に
切り替わる)が、Redis 版は TTL 起点——つまり「**その key への最初の
リクエストから window**」でウィンドウが始まる(docs/02 演習2で「別案」として
考えたモデル)。INCR + TTL による自然で単純な実装を優先した結果で、
「ウィンドウあたり limit 回」の保証は同じだが、リセットが起きるタイミングの
意味が異なることに注意。

なお `redis.NewScript` は EVALSHA(スクリプトのSHA1だけ送る)を試み、
未ロード時だけ EVAL にフォールバックする。スクリプト本文の転送は初回だけで、
以降の通信量はコマンド1個ぶんに近い。

### INCR + EXPIRE を「別々に」送ってはいけない理由

固定ウィンドウは一見 Lua 不要に見える(`INCR` は単体で原子的)。
だが「初回だけ `EXPIRE` を付ける」が別コマンドだと、INCR 直後にアプリが
落ちた場合 **TTL のないキーが永久に残る**。そのキーのカウントは二度と
リセットされず、その key は永久にブロックされ続ける。Lua で1つにまとめる
ことで「カウントしたのに期限がない」状態を構造的に排除している。

## 3. 核心②: 時刻はどこから取るか —— クロックスキュー

スライディングウィンドウとトークンバケットは時刻を使う。誰の時計を使うか:

- **アプリサーバーの時計**: サーバー間で時計がずれていると(NTPでも数ms〜、
  設定ミスなら数秒)、ずれた分だけ判定が歪む。「app1 経由なら通るのに
  app2 経由だと拒否される」が起きる
- **Redis サーバーの時計**(採用): スクリプト内で `redis.call('TIME')`。
  全サーバーが同じ時計を見るので、スキューが構造的に消える

```lua
local t = redis.call('TIME')                      -- {秒, マイクロ秒}
local now = t[1] * 1000 + math.floor(t[2] / 1000) -- ミリ秒に変換
```

副産物としてテストも楽になる。miniredis の `SetTime` でサーバー時刻を
動かせるので、インメモリ版の fakeClock と同じように決定的なテストが書ける
(Redis版リミッターに `Clock` を注入する必要がない——時刻の所有者が
Redis に移ったから)。

1点だけ残る妥協が `Result.ResetAt`。判定と `RetryAfter`(相対時間)は
Redis 時刻基準だが、`ResetAt` は絶対時刻なのでアプリ側の `time.Now()` に
相対値を足して作っている。アプリと Redis の時計がずれている環境では、
この絶対時刻(`X-RateLimit-Reset` ヘッダに使う値)だけがずれうる。
許可/拒否と `Retry-After` の正しさには影響しない。

## 4. 核心③: 掃除は TTL に任せる

インメモリ版で書いたジャニターは Redis 版には無い。代わりに全スクリプトが
状態更新のたびに `PEXPIRE` を打つ。TTL の長さは**インメモリ版の
deleteExpired と同じ論理**で決まっている:

| 実装 | TTL | 理由(インメモリ版の対応物) |
|---|---|---|
| FixedWindow | window | ウィンドウが終われば無意味 |
| SlidingWindow | 2 × window | curr は次ウィンドウの prev として生きる |
| TokenBucket | 満タンまでの時間 | 満タン = 新規キーと区別不能 |

「いつ消してよいか」の答えはアルゴリズムの性質から導かれるもので、
どこに実装するか(ジャニター vs TTL)とは独立——Stage 3・4 で考えた
ことがそのまま移植できる。

## 5. 核心④: Redis が落ちたら

分散版の新しい現実: **バックエンドは普通に落ちる**。ネットワーク断、
Redis 再起動、フェイルオーバー、タイムアウト。

本パッケージの態度は「**エラーを握りつぶさず返す。判断は呼び出し側**」。
`TestFixedWindow_RedisDownReturnsError` が示すとおり、Redis 不達なら
`Allow` はエラーを返し、Stage 6 のミドルウェアの `FailurePolicy` が
fail-open(通す)/ fail-close(503)を決める。レイヤーの責務分離:
リミッターは事実を報告し、ポリシーはHTTP層が持つ。

実務ではさらに次を検討する(本実装ではスコープ外、演習参照):

- **タイムアウトを短く**: リミッター判定のためにリクエスト全体を
  遅らせない(`redis.Options` の `DialTimeout`/`ReadTimeout` を数十msに)
- **フォールバック**: Redis 障害中だけローカルのインメモリリミッターに
  切り替える(精度は落ちるが無制限よりよい)

## 6. その他の実務的な論点

- **レイテンシ**: 判定のたびに Redis へ1往復(同一DC内で ~0.5ms)。
  超高トラフィックではこれ自体が課題になり、「ローカルで概算しつつ
  定期的に Redis と同期する」ハイブリッド構成が使われる(精度と引き換え)
- **Redis Cluster**: 1判定で触るキーを1個に収めてある(スライディング
  ウィンドウも3フィールドの**1ハッシュ**)。複数キーに分けると Cluster では
  別ノードに配置されて Lua から原子的に触れない
- **選ばなかった実装**: スライディングウィンドウ「ログ」の Redis 版は
  Sorted Set(`ZADD`/`ZREMRANGEBYSCORE`/`ZCARD`)で素直に作れるが、
  key × limit 個のメンバーを持つメモリコストも Stage 2 と同様に付いてくる。
  また、トークンバケットの兄弟である **GCRA** は状態が「次に許可できる
  理論時刻(TAT)」の**1値だけ**で済み、Redis 界隈で人気
  (go-redis 公式の redis_rate がこれ)。トークンバケットが理解できていれば
  GCRA は「残量ではなく時刻で持つ」だけの違い

## 7. 使い方

```go
client := redis.NewClient(&redis.Options{Addr: "localhost:6379"})

lim, err := redislimit.NewTokenBucket(client, 100, time.Minute,
	redislimit.WithBurst(20))
// あとはインメモリ版と完全に同じ:
mw, _ := middleware.New(lim, middleware.KeyByIP,
	middleware.WithFailurePolicy(middleware.FailOpen))
```

## 8. 演習(理解の確認)

1. 素朴な GET→判定→INCR 方式を実際に書いて、ゴルーチン100本で叩き、
   limit を超えて許可されることを確認してみる(競合の実感)
2. スライディングウィンドウ「ログ」の Redis 版を Sorted Set で実装してみる。
   `ZREMRANGEBYSCORE` → `ZCARD` → `ZADD` → `PEXPIRE` を Lua にまとめる
3. 「Redis 障害中はインメモリ版にフォールバックする Limiter ラッパー」を
   設計してみる。復旧をどう検知するか、フォールバック中の limit は
   何分の1にすべきか(ヒント: サーバー台数)
4. cmd/demo に `-algo redis-token` を追加して、実際の Redis(または
   `docker run redis`)で2プロセス起動し、制限が共有されることを curl で確認する

## 9. 全体のまとめ

これで要件定義に掲げた全ステージが完成した。振り返ると:

- **Stage 1–5**: 5つのアルゴリズム。トレードオフの軸は「正確さ・メモリ・
  バースト特性」で、速度はどれも問題にならない(~250ns)
- **Stage 6**: アルゴリズムと運用判断(key・ヘッダ・障害時)の分離
- **Stage 7**: 定番ライブラリとの答え合わせ。単一プロセスの自己制御なら
  `x/time/rate` を使えばよい
- **Stage 8**: 分散化。Mutex→Lua、Clock→TIME、ジャニター→TTL という
  「同じ問題の置き換え」で、新しい概念は原子性と障害だけ

貫かれている設計は Stage 0 で決めた `Limiter` インターフェースであり、
「分散を見据えて `ctx` と `error` を最初から入れる」という一手が
最後まで全レイヤーを無傷で保った。インターフェースの初期設計に
時間をかける価値は、こういう形で回収される。
