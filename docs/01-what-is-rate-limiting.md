# Stage 0: レートリミットとは何か・共通設計

## 1. レートリミットとは

レートリミット(Rate Limiting)は「**一定時間内に処理するリクエスト数を制限する**」仕組み。
たとえば「1ユーザーにつき1分間に60回まで」のように制限する。

### なぜ必要か

| 目的 | 例 |
|------|-----|
| 過負荷からの保護 | 一部のクライアントの暴走やスパイクでサービス全体が落ちるのを防ぐ |
| 悪用の防止 | ブルートフォース攻撃、スクレイピング、スパム投稿の抑制 |
| 公平性 | 特定ユーザーがリソースを独占しないようにする |
| コスト管理 | 従量課金の外部API呼び出しを予算内に収める |
| 商品としての制限 | 「無料プランは100回/日、有料プランは10,000回/日」のようなプラン設計 |

実例: GitHub API は認証済みで 5,000回/時、Stripe API は 100回/秒(ライブモード)など、
公開APIはほぼ例外なくレートリミットを持つ。

### 制限に達したら何が起きるか

HTTP では **`429 Too Many Requests`** を返すのが標準。あわせて:

- `Retry-After` ヘッダ — 何秒後に再試行してよいか
- `X-RateLimit-Limit` / `X-RateLimit-Remaining` / `X-RateLimit-Reset` — 上限・残り回数・リセット時刻
  (IETF で `RateLimit-*` として標準化が進行中。多くのAPIは `X-` 付きの慣習ヘッダを使っている)

を返すと、クライアントが適切にバックオフできる。

## 2. 主要アルゴリズムの全体像

これから1つずつ実装していく5つのアルゴリズムの位置づけ:

```
                  正確さ
                    ▲
    Sliding Window  │
    Log (Stage 2)   │      Token Bucket (Stage 4)
         ●          │           ● ← 実務の定番
                    │      Leaky Bucket (Stage 5)
    Sliding Window  │           ●
    Counter (Stage 3)●
                    │
         Fixed      │
         Window     ●
        (Stage 1)   │
                    └──────────────────▶ 実装の単純さ・省リソース
```

| アルゴリズム | 一言でいうと | 弱点 |
|------------|------------|------|
| 固定ウィンドウ | 「毎分0秒でカウンタをリセット」 | 境界で最大2倍通る |
| スライディングウィンドウログ | 「直近60秒のリクエストを全部記録して数える」 | メモリを食う |
| スライディングウィンドウカウンタ | 「前のウィンドウのカウントを按分して近似」 | 近似なので厳密ではない |
| トークンバケット | 「一定速度で貯まるトークンを消費して通る」 | 設定によりバーストが大きくなる |
| リーキーバケット | 「バケツから一定速度で漏れる分だけ通す」 | バーストを許容しない(用途による) |

## 3. 共通設計([ratelimit.go](../ratelimit/ratelimit.go))

全アルゴリズムに共通のインターフェースを最初に決めておく。これにより
「アルゴリズムを差し替えても呼び出し側のコードは変わらない」状態を作る。

### 3.1 Limiter インターフェース

```go
type Limiter interface {
	Allow(ctx context.Context, key string) (Result, error)
}
```

設計判断のポイント:

**なぜ `key string` を取るのか。**
レート制限は普通「ユーザーごと」「IPアドレスごと」にかける。リミッター自体は
key が何を意味するかを知らず、呼び出し側が `userID` や `clientIP` を渡す。
この分離により同じ実装を認証API・公開APIどちらにも使える。

**なぜ `bool` だけでなく `Result` を返すのか。**
実務では「拒否した」だけでは足りない。クライアントに `Retry-After` や残り回数を
伝える必要があるため、判定時に得られる情報をまとめて返す。

```go
type Result struct {
	Allowed    bool          // 許可されたか
	Limit      int           // 設定上限
	Remaining  int           // 残り回数
	RetryAfter time.Duration // 拒否時: 次に許可されるまでの待ち時間
	ResetAt    time.Time     // 制限がリセットされる時刻
}
```

**なぜ `ctx context.Context` と `error` を返すのか。インメモリなら不要では?**
その通り、インメモリ実装ではエラーはほぼ発生しない(context キャンセルのみ)。
しかし Stage 8 で作る Redis 版はネットワーク I/O を伴い、タイムアウトも障害も起きる。
**あとからインターフェースにパラメータを足すのは破壊的変更**なので、分散版を見据えて
最初から `ctx` と `error` を含めておく。これは「インターフェースは最初の設計が肝心」
という Go 実務での重要な知見。

### 3.2 Clock の注入

```go
type Clock interface {
	Now() time.Time
}
```

レートリミッターは本質的に「時間」を扱うコード。`time.Now()` を直接呼ぶと
「1分後にリセットされること」のテストに実際に1分待つ羽目になる。

そこで時刻の取得を `Clock` インターフェースに切り出し、本番では `time.Now` を使う
`SystemClock()` を、テストでは手で時間を進められる偽クロックを注入する:

```go
// テストコードから(clock_test.go)
clock := newFakeClock()
fw, _ := NewFixedWindow(2, time.Minute, WithClock(clock))
// ... 2回使い切る ...
clock.Advance(time.Minute) // 1分待つ代わりに1行
// → リセットされていることを即座に検証できる
```

これは時間依存コード全般に使える定石パターン。テストが**決定的**(何回実行しても
同じ結果)かつ**高速**になる。

### 3.3 ゴルーチンセーフであること

Web サーバーではリクエストごとにゴルーチンが立つため、リミッターは必ず複数
ゴルーチンから同時に呼ばれる。全実装で内部状態をミューテックスで保護し、
`go test -race` で検証する。
