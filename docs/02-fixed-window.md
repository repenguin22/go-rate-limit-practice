# Stage 1: 固定ウィンドウカウンタ (Fixed Window Counter)

実装: [fixedwindow.go](../ratelimit/fixedwindow.go) / テスト: [fixedwindow_test.go](../ratelimit/fixedwindow_test.go)

## 1. アイデア

時間軸を一定幅の「ウィンドウ」に区切り、ウィンドウごとにカウンタを持つ。
一番素朴で、誰でも最初に思いつく方式。

```
limit=5, window=1分 の場合:

 00:00        00:01        00:02        00:03
   |------------|------------|------------|
   ウィンドウ1     ウィンドウ2     ウィンドウ3
   カウンタ: 5/5   カウンタ: 2/5   カウンタ: 0/5

ルール:
1. リクエストが来たら、現在時刻がどのウィンドウに属するか計算
2. そのウィンドウのカウンタが limit 未満なら +1 して許可
3. limit に達していたら拒否。次のウィンドウ開始まで待たせる
```

ポイントは**ウィンドウが壁時計に整列している**こと。window=1分なら、
どの key もウィンドウは毎分00秒に切り替わる。「最初のリクエストから1分」ではない。

## 2. 実装の読みどころ

### 2.1 ウィンドウの計算は `time.Truncate` 一発

```go
start := now.Truncate(fw.window) // 例: 00:00:37 → 00:00:00
resetAt := start.Add(fw.window)  //                 00:01:00
```

`Truncate` は時刻を指定単位に切り捨てる。これで「今どのウィンドウか」が決まる。
ウィンドウの切り替わり検出も「保存してある start と今計算した start が違うか」を
見るだけでよい:

```go
c, ok := fw.counters[key]
if !ok || !c.start.Equal(start) {
	c = &windowCounter{start: start} // 新しいウィンドウとして作り直し
	fw.counters[key] = c
}
```

タイマーやバックグラウンド処理で「毎分カウンタをリセット」する必要はない。
**次のアクセス時に遅延評価でリセットする**方が、コードも単純でリソースも使わない。

### 2.2 状態は key ごとに `map` で持つ

```go
mu       sync.Mutex
counters map[string]*windowCounter
```

`map` は Go では並行アクセスに安全ではないので、`sync.Mutex` で保護する。
クリティカルセクション(ロック中の処理)はマップ参照とカウンタ増分だけなので短く、
ベンチマーク(後述)でも単一ミューテックスで十分な性能が出る。

### 2.3 メモリリーク対策: ジャニター

一度でもアクセスした key は `map` にエントリが残る。IPアドレスを key にすると
エントリは無限に増えていく——**放っておくとメモリリーク**になる。

そこでバックグラウンドで定期的に期限切れエントリを削除するゴルーチン(ジャニター)を回す:

```go
func (fw *FixedWindow) janitor(interval time.Duration) {
	ticker := time.NewTicker(interval)
	defer ticker.Stop()
	for {
		select {
		case <-fw.stop:      // Close() で確実に止められるように
			return
		case <-ticker.C:
			fw.deleteExpired(fw.clock.Now())
		}
	}
}
```

プロダクション品質として重要なのは:

- **`Close()` で止められること。** 止める手段のないゴルーチンは「ゴルーチンリーク」に
  なる。`stopOnce sync.Once` を使い、`Close` が複数回呼ばれても panic しない
  (`close` を2回呼ぶと panic する)。
- **削除しても正しさに影響しないこと。** `deleteExpired` は「ウィンドウが終了した」
  エントリしか消さない。消された key は次のアクセスで新規ウィンドウとして
  作り直されるだけなので、判定結果は変わらない。純粋なメモリ回収である。

### 2.4 テストの読みどころ

- `newFakeClock()` + `Advance()` で実時間を待たずにウィンドウ切り替えを検証
  (`TestFixedWindow_ResetsOnNewWindow`)
- `TestFixedWindow_ConcurrentAccess`: 20ゴルーチン × 10リクエスト = 200 を
  limit=100 に叩き込み、**ちょうど100だけ**許可されることを確認。
  ロックが正しくなければ100を超える(check と increment の間に割り込まれる)
- `TestFixedWindow_BoundaryBurst`: 後述の弱点を「仕様」としてテストに固定化

## 3. 弱点: 境界バースト問題

固定ウィンドウ最大の弱点。**ウィンドウの境界をまたぐと、設定の最大2倍の
リクエストが短時間に通ってしまう。**

```
limit=10/分 のはずが……

 00:00:00                00:00:59│00:01:00                00:01:59
   |                          ▼▼▼│▼▼▼                          |
   |            ここで10リクエスト│ここで10リクエスト             |
   |──────── ウィンドウ1 ────────│──────── ウィンドウ2 ─────────|

→ 00:00:59〜00:01:00 の実質1秒間に20リクエストが全部通る
```

ウィンドウ1では「1分間に10回以内」を守っているし、ウィンドウ2でも守っている。
しかし**任意の連続する60秒間**で見ると20回通っている。「1分間に10回」の直感的な
意味を破っている。

これは `TestFixedWindow_BoundaryBurst` で実証している。バックエンドの容量設計を
「最大 limit×2 のバーストが来うる」前提で行うなら固定ウィンドウでも実用になるが、
厳密に流量を守りたい場合は不適切。この問題を解決するのが次の
スライディングウィンドウ系アルゴリズム(Stage 2, 3)。

## 4. 特性まとめ

| 観点 | 評価 |
|------|------|
| 時間計算量 | O(1) / リクエスト |
| 空間計算量 | O(アクティブな key 数)。key あたり時刻+intのみで最小 |
| 正確さ | ウィンドウ内では厳密。ただし境界バーストで最大2倍 |
| 実装難度 | 最も簡単 |
| 向いている場面 | 「日次1万回まで」のようなクォータ制。多少のバーストを許容できるAPI |
| 向かない場面 | バックエンドを厳密な流量以下に守りたい場合 |

補足: Redis と相性がよい(`INCR` + `EXPIRE` だけで分散版が作れる)ため、
シンプルな分散レートリミットとして今でも広く使われている。Stage 8 で実装する。

## 5. 演習(理解の確認)

1. `TestFixedWindow_BoundaryBurst` の `limit` や `Advance` の値を変えて、
   どういうタイミングなら2倍まで通るのか手を動かして確認してみる
2. ウィンドウを壁時計に整列させず「key ごとの最初のリクエスト時刻から1分」に
   したら、境界バースト問題はどう変わるか考えてみる
   (ヒント: 問題は消えない。攻撃者は「自分のウィンドウ」の境界を知っている)
3. `go test -bench . -benchmem ./ratelimit/` を実行して、1判定あたりの
   所要時間とアロケーション数を見てみる
