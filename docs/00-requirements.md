# 要件定義: Goで学ぶレートリミット

## 1. プロジェクトの目的

レートリミット(Rate Limiting)の主要アルゴリズムを **Goで一から実装しながら** 理解する学習プロジェクト。
ただしコードは「学習用のおもちゃ」ではなく **プロダクションレディな品質** で書く。
実務でそのまま参考にできる設計・エラー処理・テストを備えることを目標とする。

- 読者(学習者)は Go の基本文法は理解している前提
- レートリミットの知識はゼロからでも読み進められる構成にする
- 「動くコード」と「なぜそう書くのか」を解説するドキュメントをセットで提供する

## 2. 学ぶこと(スコープ)

### 2.1 実装するアルゴリズム(学習ステージ順)

| Stage | アルゴリズム | 学びのポイント |
|-------|------------|--------------|
| 1 | 固定ウィンドウカウンタ (Fixed Window Counter) | 最も単純な発想。境界バースト問題を体感する |
| 2 | スライディングウィンドウログ (Sliding Window Log) | 正確だがメモリを食う。トレードオフを学ぶ |
| 3 | スライディングウィンドウカウンタ (Sliding Window Counter) | 近似で精度とコストのバランスを取る |
| 4 | トークンバケット (Token Bucket) | 実務で最も使われる方式。バースト許容の考え方 |
| 5 | リーキーバケット (Leaky Bucket) | 流量の平滑化。トークンバケットとの違い |

### 2.2 共通設計

- 全アルゴリズムが満たす **共通インターフェース** を最初に定義する
  - `Allow(ctx context.Context, key string) (Result, error)` を中心とした最小のインターフェース
  - 分散版はネットワークI/Oを伴うため、最初から `context.Context` とエラーを返す設計にする(インメモリ版もこれに合わせる)
  - `Result` には許可/拒否だけでなく `Limit` / `Remaining` / `RetryAfter` / `ResetAt` を含め、HTTPレスポンスヘッダにそのまま使えるようにする
- key 単位(ユーザーID・IPアドレスなど)で制限できること
- 時刻は `time.Now()` 直呼びせず **クロックを注入可能** にする(テスト容易性のため)

### 2.3 応用ステージ

| Stage | 内容 | 学びのポイント |
|-------|------|--------------|
| 6 | HTTPミドルウェア | `net/http` への組み込み。429 / `Retry-After` / `RateLimit-*` ヘッダ。バックエンド障害時の fail-open / fail-close |
| 7 | 標準準拠ライブラリとの比較 | `golang.org/x/time/rate` を読み、自作実装と比較する |
| 8 | **分散レートリミット (Redis)** | 複数インスタンス間で制限を共有する。実務ではほぼ必須。Luaスクリプトによる原子性、Redisサーバー時刻の利用(クロックスキュー対策)、TTLによる自動クリーンアップ、障害時のフォールバック戦略 |

Stage 8 では以下を Redis バックエンドで実装する:

- 固定ウィンドウ(`INCR` + `EXPIRE`)— 分散版の入門
- スライディングウィンドウカウンタ — 精度とコストのバランス
- トークンバケット(Luaスクリプト)— GCRA の考え方にも触れる

### 2.4 スコープ外(将来の発展課題として言及のみ)

- gRPC インターセプタ対応
- 動的な設定変更・設定ファイルの読み込み
- Redis Cluster / マルチリージョン構成での厳密な一貫性

## 3. 機能要件

- **FR-1**: 各アルゴリズムは key ごとに独立したレート制限を行う
- **FR-2**: 制限値(回数・期間・バーストサイズ等)はコンストラクタで設定でき、不正な値はエラーで拒否する
- **FR-3**: 判定結果 `Result` は許可/拒否に加え、残り回数・リセット時刻・次に許可されるまでの待ち時間を返す
- **FR-4**: HTTPミドルウェアは拒否時に `429 Too Many Requests` と `Retry-After` を返し、`RateLimit-*` 系ヘッダを付与する
- **FR-5**: 使われなくなった key の状態を掃除できる(メモリリーク対策。インメモリ版はジャニター、Redis版はTTL)
- **FR-6**: Redis 版は複数プロセスから同時に使っても正しく制限を共有する(check-and-set の競合が起きない = Lua による原子的更新)
- **FR-7**: Redis 版はバックエンド障害時にエラーを呼び出し元へ返し、ミドルウェア側で fail-open(通す)/ fail-close(拒否)を選択できる

## 4. 非機能要件(プロダクション品質の定義)

- **NFR-1(並行安全)**: すべての実装はゴルーチンセーフであること。`go test -race ./...` をパスする
- **NFR-2(テスト)**: 各アルゴリズムにテーブル駆動テストを書く。注入クロックで時間経過を決定的にテストする。Redis 版は miniredis によるテストを備える
- **NFR-3(ベンチマーク)**: 各アルゴリズムに `Benchmark` を用意し、性能特性を比較できるようにする
- **NFR-4(依存の最小化)**: インメモリ実装・ミドルウェアは標準ライブラリのみ。外部依存は Redis クライアント(`github.com/redis/go-redis/v9`)、テスト用 `miniredis`、比較用 `golang.org/x/time/rate` に限定する
- **NFR-5(ドキュメント)**: 各 Stage に対応する解説ドキュメント(日本語)を用意する。全公開識別子に godoc コメントを付ける
- **NFR-6(エラー処理)**: エラーを握りつぶさない。呼び出し元が判定に使えるエラーは公開エラー変数として定義する
- **NFR-7(リソース管理)**: ゴルーチンを起動する実装は `Close()` で確実に停止できる(ゴルーチンリークを作らない)
- **NFR-8(静的検査)**: `go vet ./...` をクリーンに保つ

## 6. 進め方

1 Stage ずつ「実装 → テスト → 解説ドキュメント」の順で完成させて進む。
一度に全部作らず、各 Stage を読んで理解してから次へ進める。
