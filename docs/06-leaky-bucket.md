# Stage 5: リーキーバケット (Leaky Bucket)

実装: [leakybucket.go](../ratelimit/leakybucket.go) / テスト: [leakybucket_test.go](../ratelimit/leakybucket_test.go)

## 1. アイデア: 底に穴の空いたバケツ

リクエストのたびにバケツへ水を1単位注ぐ。バケツの底には穴が空いていて、
window あたり limit 単位の一定速度で漏れ続ける。注ごうとして溢れるなら拒否。

```
    リクエスト ──▶ 水を1単位注ぐ。溢れるなら拒否
                      │
                      ▼
              ┌───────────────┐
              │ ~~~~~~~~~~~~~ │ ← 水位(容量 = capacity まで)
              │ ~~~~~~~~~~~~~ │
              └───────┬───────┘
                      ▼ 漏れ: window あたり limit の一定速度
```

```
ルール:
1. 水は window あたり limit 単位の一定速度で漏れ続ける
2. リクエストが来たら水位+1。ただし容量を超えるなら拒否
3. 水位が下がった分だけ、また注げるようになる
```

## 2. 種明かし: トークンバケットと同じ数式である

まず正直に言うと、この「メーター型」リーキーバケットは
**トークンバケットの鏡像**にすぎない。`tokens = capacity − level` と置き換えると:

| | TokenBucket | LeakyBucket |
|---|---|---|
| 状態 | トークン残量(多いほど余裕) | 水位(少ないほど余裕) |
| 時間経過 | トークンが**増える** | 水位が**減る** |
| リクエスト | トークンを1個**消費** | 水を1単位**追加** |
| 上限/下限 | 容量で頭打ち | 0で底打ち |
| 拒否条件 | `tokens < 1` | `level + 1 > capacity` |

実装を並べて読むと、`math.Min` と `math.Max`、`+` と `−` が入れ替わって
いるだけなのが分かる。片方を理解していればもう片方はタダで手に入る。
テストも意図的に対称に書いてある(`SteadyRefill` ⇔ `SteadyLeak`、
`RefillCapsAtBurst` ⇔ `DrainsToEmpty`)。

**では何が違うのか。** 数式ではなく、**典型的な使い方**が違う。

### 2.1 初期状態の非対称性

数式は双対でも、初期状態だけは対称でない:

- トークンバケット: 満タンから始まる → 新規クライアントは**いきなり burst 回使える**
- リーキーバケット: 空から始まる → 新規クライアントも容量までしか注げない
  (結果は同じだが、「使っていなかった分の貯金」という物語がない)

### 2.2 容量の意味づけの違い

- トークンバケットの容量(burst)は「**許すバースト量**」。大きめに設定して
  クライアントの利便性を上げる方向で使う
- リーキーバケットの容量は「**許容する待ち行列の深さ**」。小さめに設定して
  後段への流量を守る方向で使う

極端な設定が両者の性格を最もよく表す。**容量=1のリーキーバケット**は
「リクエスト間隔は最低 window/limit」という完全平滑化になる:

```go
// 10回/分・容量1 → どんなに頑張っても6秒に1回しか通らない
lb, _ := ratelimit.NewLeakyBucket(10, time.Minute, ratelimit.WithBurst(1))
```

`TestLeakyBucket_CapacityOneSmoothing` がこれを検証している。バースト0、
完全に等間隔——外部APIのレート契約(「秒間10コールまで」)を1コールも
超えたくない場合などに使う。

### 2.3 「キュー型」リーキーバケット —— もうひとつの姿

本来のリーキーバケット(ネットワーク機器のトラフィックシェーピング由来)は
**拒否せずに並ばせる**:

```
リクエスト ──▶ [キュー(バケツ)] ──▶ 一定速度で取り出して処理
                 満杯なら拒否            (漏れ = 処理)
```

- メーター型(本実装): 溢れたら**拒否**する。ポリシング(policing)
- キュー型: 溢れない限り**待たせて**、出力を完全に一定速度にする。
  シェーピング(shaping)

キュー型は `Allow() bool` ではなく「処理してよくなるまでブロックする
`Wait(ctx)`」というAPIになる。`x/time/rate` の `Wait` メソッドがまさに
これで、Stage 7 で読む。

## 3. 実装の読みどころ

Stage 4 を読んでいれば新しい概念はほぼない。差分だけ:

- 遅延計算が「補充」ではなく「漏れ」: `level = max(0, level − leakedIn(elapsed))`
- RetryAfter は「溢れる分が漏れるまで」: `durationFor(level + 1 − capacity)`
- 掃除の条件は「空に戻ったか」(トークンバケットの「満タンに戻ったか」と対称):

```go
if st.level-lb.leakedIn(now.Sub(st.last)) <= 0 {
	delete(lb.states, key)
}
```

浮動小数点の「除算は最後に」も Stage 4 と同じ理由で適用済み。

## 4. 特性まとめ(バケット2方式の使い分け)

| 観点 | TokenBucket | LeakyBucket(メーター型) |
|------|-------------|--------------------------|
| 数式 | 双対(同じ) | 双対(同じ) |
| 設計思想 | バーストを**許す** | 流量を**平滑化する** |
| 典型的な容量設定 | 大きめ(= burst 許容) | 小さめ(= 等間隔強制) |
| 1判定の速度 (M1 Pro) | ~290ns | ~277ns |
| 向いている場面 | 受信側の一般API保護 | 送信側の外部API呼び出し制御、後段保護 |

覚え方: **守るときはトークンバケット、守らせるときはリーキーバケット**。
自分のAPIを守るなら利用者にバーストの自由を与えたい(トークン)。
他社のAPI契約を守るなら1回も超えたくない(リーキー、容量小さめ)。

## 5. 演習(理解の確認)

1. `TokenBucket` + `WithBurst(1)` と `LeakyBucket` + `WithBurst(1)` の挙動を
   テストで比較してみる。初期状態の1回分以外は同じになるはず
2. キュー型(`Wait(ctx)` でブロックする版)のシグネチャと内部設計を考えてみる。
   拒否型と違って「何個まで待たせるか」という新しいパラメータが必要になる
3. 5つのアルゴリズムが出揃った。「ログイン試行の制限」「社内バッチから
   外部APIへの呼び出し」「公開APIの無料プラン」それぞれにどれを選ぶか、
   理由付きで決めてみる

## 6. ここまでの総括と次のステージ

5方式の全体マップ(Stage 0 の図の完成版):

| | 状態サイズ | 保証 | バースト | 代表用途 |
|---|---|---|---|---|
| FixedWindow | 最小 | ウィンドウ内厳密 | 境界で2倍漏れる | 粗いクォータ |
| SlidingWindowLog | O(limit) | **厳密** | なし | 高価値操作 |
| SlidingWindowCounter | 最小×2 | 近似 | ほぼなし | 汎用(分散向き) |
| TokenBucket | 最小 | 平均レート厳密 | **設定可能** | 汎用(実務の第一候補) |
| LeakyBucket | 最小 | 平均レート厳密 | 抑制(平滑化) | 送信側制御 |
