# Stage 2: スライディングウィンドウログ (Sliding Window Log)

実装: [slidingwindowlog.go](../ratelimit/slidingwindowlog.go) / テスト: [slidingwindowlog_test.go](../ratelimit/slidingwindowlog_test.go)

## 1. アイデア: ウィンドウを「固定」せず「滑らせる」

固定ウィンドウの境界バーストの原因は、ウィンドウが壁時計に固定されていて
**境界をまたぐとカウントが突然ゼロに戻る**ことだった。

ならば発想を変えて、判定のたびに「**現在時刻から window だけ遡った範囲**」を
ウィンドウとして使えばいい。ウィンドウは常に現在時刻とともに滑る(スライドする)。

```
limit=10, window=1分。00:01:20 にリクエストが来たら:

              ┌──── この範囲のリクエスト数を数える ────┐
  ────────────┤                                        │
              00:00:20                            00:01:20(いま)
                   ↑ 常に「いま」から60秒遡る。境界は存在しない
```

これを実現するには「範囲内に何件あったか」を数えられる必要がある。
そこで **許可したリクエストの時刻を全部記録しておく**。これが「ログ」の意味。

```
ルール:
1. リクエストが来たら、ログから (now - window) 以前の記録を捨てる
2. 残った記録が limit 未満なら、now をログに追加して許可
3. limit 件あるなら拒否
```

任意の連続する window 幅を切り取っても limit 以下 —— これを**厳密に**保証できる。
5つのアルゴリズムの中で唯一、近似なしの正確な方式。

## 2. 実装の読みどころ

### 2.1 状態は「時刻の配列」

```go
logs map[string][]time.Time // 各 key の許可済みリクエスト時刻(昇順)
```

固定ウィンドウの状態は key あたり `(開始時刻, カウント)` の2値だったが、
今回は**タイムスタンプの列**を持つ。append しかしないので自然に昇順が保たれる。

### 2.2 刈り込みは先頭から見るだけ

```go
cutoff := now.Add(-sl.window)
drop := 0
for drop < len(log) && !log[drop].After(cutoff) {
	drop++
}
log = log[drop:]
```

ログが昇順であることを利用すると、期限切れの記録は必ず**先頭に固まっている**。
先頭から「cutoff 以前」の要素を数えて切り落とすだけでよい。
`!t.After(cutoff)` は `t <= cutoff` の意味(`time.Time` に `<=` はないのでこう書く)。

境界の扱いを1か所に固定しているのもポイント: 時刻 `t` の記録は
`t + window` の瞬間に**ちょうど期限切れになる**。この一貫性があるから、
拒否時の `RetryAfter` を正確に計算できる(次項)。

### 2.3 RetryAfter が「正確に」出せる

固定ウィンドウの `RetryAfter` は「次のウィンドウ開始まで」だったが、
この方式では**最古の記録がいつ期限切れになるか**がそのまま答えになる:

```go
freeAt := log[0].Add(sl.window) // 最古の記録が消える瞬間 = 1枠空く瞬間
```

`TestSlidingWindowLog_RetryAfterPointsToOldestExpiry` で検証している。
枠は一気にリセットされるのではなく、**古い記録から1件ずつ順に回復していく**。
この挙動は `TestSlidingWindowLog_PartialExpiry` が分かりやすい:
00:00:00 に5件 + 00:00:30 に5件で満杯 → 00:01:00 には最初の5件だけ期限切れ
→ ちょうど5枠だけ回復する。

### 2.4 境界バーストが消えたことをテストで確認する

`TestSlidingWindowLog_NoBoundaryBurst` は Stage 1 の
`TestFixedWindow_BoundaryBurst` と**同じシナリオ**を実行する:

- 00:00:59 に10回 → 許可(ここまでは同じ)
- 00:01:00 にさらに10回 → 固定ウィンドウは全部通したが、**今回は全部拒否**。
  直近60秒(00:00:00〜00:01:00)にすでに10件の記録があるから

同じ攻撃シナリオを両方のテストに固定化してあるので、2つのファイルを
並べて読むとアルゴリズムの違いがそのまま見える。

### 2.5 メモリ回収は2段構え

- **Allow 内の刈り込み**: アクセスされた key は判定のついでに掃除される
- **ジャニター**: アクセスが途絶えた key はログが残り続けるので、
  「最新の記録すら期限切れ」になった key を定期削除する

```go
// ログは昇順なので、最新(末尾)が期限切れなら全件期限切れ。
if len(log) == 0 || !log[len(log)-1].After(cutoff) {
	delete(sl.logs, key)
}
```

## 3. 弱点: メモリと計算量

正確さの代償は**リソース消費**。

- **メモリ**: key あたり最大 limit 個の `time.Time`(1個24バイト)。
  「100万ユーザー × limit 1000」なら最悪 24GB —— 現実的でない。
  固定ウィンドウなら同条件で数十MBで済む
- **計算量**: 刈り込みは1判定あたり O(期限切れ件数)。ならし O(1) だが、
  タイムスタンプの append と再スライスでアロケーションも発生する

ベンチマーク(M1 Pro)では単発の速度差は小さいが、これは1 keyの話。
本当の差は key 数 × limit に比例するメモリで効いてくる。

| | FixedWindow | SlidingWindowLog |
|---|---|---|
| ns/op | ~234 | ~261 |
| key あたりのメモリ | 定数(時刻+int) | O(limit) |

## 4. 特性まとめ

| 観点 | 評価 |
|------|------|
| 時間計算量 | ならし O(1)、最悪 O(limit) |
| 空間計算量 | O(key 数 × limit) — **これが問題** |
| 正確さ | 厳密。任意の連続 window で limit 以下を保証 |
| 実装難度 | 簡単 |
| 向いている場面 | limit が小さい高価値操作(ログイン試行 5回/分、SMS送信 3回/時など) |
| 向かない場面 | limit が大きい一般APIの流量制限 |

「正確さが必要な場面では limit も小さいことが多い」ため、実務でも
用途を選べば現役の方式。Redis の Sorted Set を使った分散版も素直に作れるが、
本プロジェクトの Stage 8 では実装していない(Stage 8 の演習2として扱う)。

## 5. 演習(理解の確認)

1. `TestSlidingWindowLog_PartialExpiry` の時刻をずらして、枠が「1件ずつ」
   回復する様子を確認してみる(例: 10件を6秒間隔で入れたらどうなるか)
2. ログを `[]time.Time` ではなくリングバッファで実装したら何が改善されるか
   考えてみる(ヒント: 再スライスと append の再確保がなくなる)
3. 「記録するのは許可したリクエストだけ。拒否したリクエストも記録する設計に
   したら挙動はどう変わるか?」を考えてみる
   (ヒント: 拒否も数えると、制限超過中にリトライし続けるクライアントは永遠に許可されない)
